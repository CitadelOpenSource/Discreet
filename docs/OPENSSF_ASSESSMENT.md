## 1. Signed Releases

**Status: PLANNED**

Releases are currently published as git tags on GitHub. Binary artifacts
are not yet cryptographically signed.

**Current state:**
- Git tags created for each release
- GitHub Actions CI builds artifacts
- No GPG or Sigstore signing of release binaries
- No SLSA provenance attestations

**Gap:**
- Release binaries should be signed with Sigstore cosign or GPG
- SLSA Level 2 provenance should be generated by CI
- Checksums (SHA-256) should be published alongside each release

**Plan:**
- Add `cosign sign-blob` step to release workflow
- Generate SLSA provenance via `slsa-framework/slsa-github-generator`
- Publish SHA-256 checksums in release notes
- Target: before v1.0 release

## 2. Branch Protection

**Status: ENABLED**

The `main` branch has protection rules enforced via GitHub.

**Current state:**
- Require pull request reviews before merging
- Require status checks to pass (CI build, clippy, tests)
- No direct pushes to main
- Force push disabled on main
- Signed commits not required (planned)

**Gap:**
- Enable "Require signed commits" once team has GPG/SSH signing configured
- Consider requiring 2 reviewers for security-sensitive paths (`src/citadel_auth*`, `src/citadel_crypto*`)

## 4. Fuzzing

**Status: IMPLEMENTED**

Property-based fuzz testing via proptest covers input validation and crypto primitives.

**Current state:**
- `proptest = "1"` in dev-dependencies
- Fuzz targets in `src/discreet_input_validation.rs`:
  - Username control char rejection (1000 cases)
  - Message length enforcement (500 cases each for valid/invalid)
  - Email format rejection without @ (500 cases)
  - Base64 encode/decode roundtrip (500 cases)
  - Rate limit key generation no-panic (500 cases)
- Total: 3000+ generated test cases per run

**Gap:**
- No continuous fuzzing infrastructure (e.g., OSS-Fuzz, ClusterFuzzLite)
- No fuzz targets for WebSocket message parsing
- No fuzz targets for MLS key package handling
- Consider `cargo-fuzz` with libfuzzer for deeper coverage

## 5. Static Analysis (SAST)

**Status: ENFORCED**

Clippy with `-D warnings` is required. All warnings are treated as errors.

**Current state:**
- `cargo clippy -- -D warnings` must pass before every commit (per CLAUDE.md)
- CI workflow runs clippy on every push and PR
- `cargo check` and `cargo test --lib` in CI pipeline
- Frontend: `tsc --noEmit` type checking, `npm run build` must pass
- No `unsafe` blocks in application code (only in dependencies)
- Compile-time SQL validation via `sqlx::query!` macros

**Gap:**
- No CodeQL or Semgrep integration for deeper semantic analysis
- No frontend SAST (consider eslint-plugin-security)
- Consider `cargo-geiger` to audit unsafe code in dependencies

## 7. Security Policy

**Status: PUBLISHED**

`SECURITY.md` is comprehensive and covers disclosure, architecture, and self-assessment.

**Current state:**
- `SECURITY.md` at repository root
- Vulnerability reporting: security@discreetai.net
- Private security advisory via GitHub
- 24-hour acknowledgment target, 72-hour patch target for critical issues
- Detailed encryption architecture documentation
- OWASP Top 10 self-assessment (91/100)
- Known advisories section with tracking
- Self-hosting security guidance
- Supply chain security section
- `docs/THREAT_MODEL.md` for security researchers

## 8. Known Vulnerabilities

**Status: TRACKED**

`cargo audit` and `npm audit` run in CI and track all known advisories.

**Current state:**
- `.github/workflows/security-audit.yml` runs weekly and on every push/PR
- `cargo audit` checks Rust dependencies against RustSec advisory database
- `npm audit --audit-level=high` checks JavaScript dependencies
- Known advisories documented in `SECURITY.md` with impact assessment
- OpenMLS 0.5 curve25519-dalek CVE documented as known issue with upgrade plan

**Gap:**
- `.cargo/audit.toml` not yet created for formally acknowledging known advisories
- No automated issue creation when new CVEs are discovered
- OpenMLS upgrade to 0.8.x still pending (blocks production deployment)
